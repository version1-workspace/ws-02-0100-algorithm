# プロジェクト設計

このリポジトリは、アルゴリズム問題の実装と、それを実行する runner を分けて管理します。

問題ごとのファイル構成をそろえることで、問題を追加しても CLI 側の仕組みを変えずに検出・実行できる状態を保ちます。

## ディレクトリの役割

- `problems/`: アルゴリズム問題をカテゴリ別に配置する領域
- `cli/`: `problems/` 配下の問題ディレクトリを検出し、解答とテストケースを読み込んで実行する runner
- `docs/`: プロジェクト全体の設計、運用ルール、問題追加ルールなど、コードだけでは分かりにくい背景を残す領域

設問ディレクトリの命名や必須ファイルの配置ルールは、[設問ディレクトリの配置ルール](problems/design.md) に分けています。

## 問題ディレクトリの考え方

各問題は `problems/{category}/{problem}/` の単位で管理します。

```text
problems/
└── NN-category-name/
    └── MMM-problem-name/
        ├── answer.ts
        ├── solution.ts
        ├── testcase.ts
        ├── question.md
        └── lib.ts
```

問題ごとにファイルを分ける意図は、学習者が触る実装、検証データ、問題文、補助コードを混ぜないためです。

- `answer.ts`: 学習者の解答実装を置く
- `solution.ts`: 模範解答を置く
- `testcase.ts`: runner が読み込むテストケースを置く
- `question.md`: 問題文と制約を置く
- `lib.ts`: その問題で使う補助関数やクラスを置く

`lib.ts` は問題固有の補助コードを置く場所です。複数問題で共有する前提の仕組みは、問題ディレクトリへ閉じ込めず、別途責務を見直します。

## runner と問題実装の責務分離

`cli/` は、どの問題を実行するかを決め、`answer.ts` または `solution.ts` と `testcase.ts` を読み込んで結果を表示します。

一方で、各問題ディレクトリは「解く対象」と「その検証データ」だけを持ちます。問題実装側が CLI の出力形式や探索方法を知る必要はありません。

この分離により、問題を追加するときは `problems/` 配下に決められたファイルを置くだけで済みます。runner の変更は、実行方法や検出ルールそのものを変える場合に限定します。

## カテゴリ追加の考え方

カテゴリは、学習順序と探索しやすさを両立するために `NN-category-name` 形式で追加します。

新しいカテゴリを追加するときは、既存カテゴリに収まらないテーマか、学習上まとめて扱う価値がある単位かを確認します。単発の問題だけのために細かくカテゴリを増やすと、全体の見通しが悪くなります。

配置ルールの詳細は [設問ディレクトリの配置ルール](problems/design.md) を参照してください。
