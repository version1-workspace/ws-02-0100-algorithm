# Solution Writer

`problems/{category}/{problem}/` 配下の `question.md` と `solution.ts` を根拠に、同じディレクトリへ `solution.md` を作成する。

## Workflow

1. 対象ディレクトリを特定する。
   - ユーザーが問題ディレクトリを指定した場合はそこを使う。
   - 指定が曖昧な場合は、該当しそうな `question.md` と `solution.ts` を探して確認する。
2. `question.md` を読み、問題の目的、入力、出力、条件、制約、例を把握する。
3. `solution.ts` を読み、採用しているアルゴリズム、主要なデータ構造、状態変数、分岐、ループ不変条件を把握する。
4. `solution.ts` の挙動と `question.md` の条件が矛盾していないか確認する。
   - 矛盾やバグの疑いがある場合は、`solution.md` を作る前にユーザーへ簡潔に報告する。
   - 明らかな typo や軽微な説明上の補足で済む場合は、解説側で前提を明示する。
5. `solution.md` を作成または更新する。

## solution.md Format

日本語で書く。基本構成は以下にそろえる。

```markdown
# 解説

## 考え方

問題を解くための発想を説明する。

## アルゴリズム

1. 手順を書く
2. 手順を書く
3. 手順を書く

## 実装のポイント

- `solution.ts` の重要な変数や関数の役割を書く
- 境界条件や注意点を書く

## 計算量

- 時間計算量: O(...)
- 空間計算量: O(...)
```

必要な場合だけ、次の節を追加する。

- `## 例で確認`: 問題例を使うと理解が大きく進む場合。
- `## なぜ正しいか`: 貪欲法、二分探索、動的計画法、グラフ探索などで正当性の説明が重要な場合。
- `## 別解`: `solution.ts` とは別の代表的な解法に触れる価値がある場合。ただし主役は `solution.ts` の解法にする。

## Writing Guidelines

- `solution.ts` に実装されていない解法を、主解法として説明しない。
- コード全体を貼り直さない。必要な識別子だけをインラインコードで参照する。
- 学習者向けに、結論だけでなく「なぜそのデータ構造や状態が必要か」を説明する。
- 問題文の制約から計算量が十分かを述べる。
- `return []` など到達しない想定のフォールバックがある場合は、問題条件との関係を説明する。
- TypeScript 固有の型や `Map`、`Set`、配列操作が重要な場合は、その役割を短く説明する。
- Markdown は簡潔に保ち、過度に長い証明やコード引用を避ける。

## Completion Checklist

- `question.md` の条件と制約に沿っている。
- `solution.ts` の実装内容と説明が一致している。
- 主要な状態変数・データ構造の役割を説明している。
- 時間計算量と空間計算量を書いている。
- `solution.md` を対象問題ディレクトリに保存している。
